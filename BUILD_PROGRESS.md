## Decisions
- Canonical code lives in `app/` (unified Vite + React client + Node WS server). The proven core
  (types/store/tools/seed/loop) is ported in from `headless-loop/`; the headless harness is preserved
  as `app/`'s regression eval (Day 7 gate). `headless-loop/` stays as the day-1/2 artifact.
- One SVG draw table is the single source of pixels (browser view == server raster). No parallel renderer.
- Demo altitude: single in-memory doc, single writer, snapshot undo, hardcoded seeds. No persistence/multiplayer.

### Post-build fix (2026-06-17)
- **"Said done but nothing showed" — FIXED.** `createFrame` defaulted to a white fill on the white page (invisible), and the draw table had no border support, so created frames (pricing section + tiers) rendered invisibly. Added `style.stroke` to the Node type (additive), border rendering in the shared draw table (`svg-build.ts`, applies to FRAME/RECT/ELLIPSE, so browser + raster stay identical), and a default light border (`#D1D5DB`) on `createFrame`. Now created frames show as clean white cards. Also softened the empty-plan escalation message (a non-design prompt like "why?" now gets a friendly "describe a design change" hint instead of "Planner returned no steps"). Verified live: pricing run renders 3 visible tier cards in a row. typecheck/build green.

### Post-build fix #2 (2026-06-17) — empty-plan flake
- **"organize the cards… → Couldn't complete" — FIXED.** The planner occasionally returned an empty `steps` array for align/distribute-of-existing-nodes prompts; the loop then escalated. `plan()` now **retries once with a hard nudge** when the first attempt is empty before giving up (5/5 + live e2e on "organize the cards horizontally with even spacing" now arrange the cards into an even top-aligned row). Also: switching the seed picker now **clears the previous doc's run history/activity** (`runStore.resetForSeed()` wired in `ws.ts` on a genuine seedDocId change) — stale "pricing section" history was showing on the scattered doc and confusing the picture. `smoke:eval` still 4/4, no regression.

### Post-build fix #3 (2026-06-17) — align-a-set couldn't verify
- **"align all the titles" → retried alignDistribute then escalated/reverted — FIXED.** Two root causes: (1) the criterion vocabulary had NO way to assert "a set of nodes share an edge", so the planner fell back to a tautological `childCount` (false-pass) or a guessed bbox `prop` (never-pass → infinite retry → escalate); (2) the planner's scoped `getTree` was depth-2, so nested nodes (card titles at depth 3) were invisible and un-referenceable. Fixes: added an `aligned` criterion (`{ids, edge, tol}` — verifies existing nodes agree on LEFT/RIGHT/TOP/BOTTOM/CENTER_X/CENTER_Y within tol) in types/verify/plan-schema with guidance ("THIS is how you verify an align/distribute step; do NOT use childCount/guessed-bbox"); deepened plan + act `getTree` depth 2→5 (cheap at demo scale) so nested elements are visible. Verified: planner now emits `{kind:'aligned', ids:[3 title ids], edge:'TOP'}` 3/3, live e2e top-aligns the titles to DONE (all y=160). `smoke:eval` still 4/4.

### Feature: user-editable canvas (2026-06-17)
- **The canvas is now directly editable by a human, not only by the agent.** Direct manipulation — drag-move, resize handles, delete, add nodes via a toolbar (frame/text/rect/ellipse), drag-to-reparent, and a property inspector (geometry / fill / typography). Human edits flow through the **same tool/op pipeline as the agent**: a new client→server `{t:'tool', name, args, baseVersion}` message dispatches against the registry and echoes the existing `ops-applied` frame (no `activityId` for human edits), so the browser is no longer a pure read-only mirror — it mutates the authoritative doc via the same boundary-validated `{ops,version}` path.
- **Editing is locked while the agent runs** (the existing single-writer guard now also rejects `{t:'tool'}` mid-run). **One-level undo** coalesces human edits with a following run (snapshot captured only when none is pending) but keeps agent runs per-run independent.
